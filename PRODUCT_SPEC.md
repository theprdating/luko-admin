## 1. 產品願景與定位

### 核心理念

**Luko** 是一款以「外在形象」為入場門檻的交友平台。
我們相信：有在認真打理自己的人，在生活態度上也更自律、更用心。

> 這不只是一個找對象的地方，而是一個**自我形象有一定水準的人才能進入的社群**。

### 一句話定位

> 「你不需要完美，但你需要認真對待自己。」

### 與現有交友軟體的差異

| 平台 | 定位 | 門檻 |
|------|------|------|
| Tinder | 大眾速配 | 無 |
| Bumble | 女性主導 | 無 |
| Hinge | 認真交往 | 無 |
| **Luko** | **形象意識社群** | **人工審核（外貌 + 打理程度）** |

## 5. 入會審核機制

### 當前方向

**人工審核**——由平台團隊審查申請者提交的照片

### 審核流程設計（草案）

```
1. 用戶提交申請
   └─ 上傳 2–3 張近期真實照片（正臉 + 半身）
   └─ 簡短自我介紹（選填）

2. 人工審核
   └─ 審核時間目標：24–48 小時內
   └─ 審核結果：通過 / 未通過 / 補件

3. 通知結果
   └─ 通過 → 開通帳號
   └─ 未通過 → 說明原因方向（不露出具體分數）
   └─ 未通過用戶可於 30 天後重新申請
```

### 審核標準（已定義）

採用**客觀底線 + 主觀綜合判斷**的兩層架構：

#### 第一層：客觀標準（必須全部通過）

```
✅ 照片清晰，能辨認臉部（非模糊、非逆光）
✅ 近期拍攝（看起來是近期照片，非童年照或明顯 10 年前）
✅ 真實本人（非偶像明星、非卡通、非 AI 生成圖）
✅ 基本打扮（非剛起床亂髮的隨手拍、有基本整潔感）
✅ 照片為本人（正臉可辨認，不全程遮臉或戴面具）
```

#### 第二層：主觀綜合判斷（通過第一層後）

```
審核員憑整體觀感判斷：
「這個人看起來有在認真對待自己嗎？」

可以通過的例子：
  ✅ 整潔穿著，拍照有基本構圖意識
  ✅ 素顏但膚況整潔、表情自然
  ✅ 運動健身照（展現自律生活）

不予通過的例子：
  ❌ 所有照片都是超重度濾鏡（無法判斷本人）
  ❌ 隨手在廁所或凌亂環境拍的快照
  ❌ 明顯無心打扮的倉促自拍

法律保護原則：
  ⚠️ 拒絕通知永遠只說「暫時未符合社群標準」
  ⚠️ 審核員不得在任何地方記錄「不夠好看」等外貌評語
  ⚠️ 審核記錄只記錄「通過/拒絕」與可選的內部備註
```

### 審核注意事項

- 需建立明確的審核 SOP，避免主觀差異過大
- 考慮建立申訴機制
- 注意不歧視特定族群（法律合規）

## 7. 技術架構

> 根據團隊現況（2人、Flutter 經驗、1個月 MVP 目標）選定的實際可執行架構。

### 團隊現況

```
人數：2 人

角色分工：
  - Co-founder A（你）
      UI/UX 設計
      Flutter 開發（iOS + Android）
      後端邏輯（Supabase）
      早期用戶審核

  - Co-founder B
      產品方向共同決策
      早期用戶審核
      營運 / 其他非技術事務

MVP 目標時程：1 個月內
審核人力：初期由兩位 Co-founder 自行審核
```

---

### 選定技術棧：Flutter + Supabase

#### 為什麼選 Supabase（而非 Firebase）

| 項目 | Supabase | Firebase |
|------|----------|----------|
| 資料庫 | PostgreSQL（關聯式）✅ | Firestore（文件式）|
| 配對邏輯查詢 | SQL，複雜查詢容易寫 ✅ | 需要大量 client 邏輯 |
| 存取控制 | Row Level Security（內建）✅ | Security Rules（較難維護）|
| 即時通訊 | Supabase Realtime ✅ | Firebase Realtime DB ✅ |
| 照片儲存 | Supabase Storage ✅ | Firebase Storage ✅ |
| 費用 | 免費額度更大 ✅ | 超量後費用較高 |

> 💡 兩者都熟的情況下，**Supabase 更適合這個專案**，主要原因是配對、審核記錄等邏輯用 SQL 寫會乾淨很多。

### Supabase 資料庫結構（草案）

```sql
-- 用戶申請表

  bio,           -- 自我介紹（選填）

swipes (
  id, from_user, to_user,
  direction,     -- like / pass

)

-- 聊天訊息
messages (
  id, match_id, sender_id,
  content, created_at, read_at
)
```

### 效能與資料庫優化

> 設計階段就要考慮，事後補救成本極高。

#### 卡片滑動（最核心的效能場景）

```
問題：用戶每次滑動都 query DB 會造成明顯延遲

解法：預載策略（Pre-fetch）
  ┌─────────────────────────────────────┐
  │  畫面上顯示 1 張卡片（當前）         │
  │  背景預載 4 張（Queue）             │
  │  每滑掉 1 張，背景自動補充 1 張      │
  └─────────────────────────────────────┘

實作：
  - 維護一個 profileQueue（長度 5）
  - 每次 queue 長度 < 3，觸發背景 fetch
  - 使用 Riverpod AsyncNotifier 管理狀態
```

#### Supabase Query 優化

```sql
-- 必建的 Index（避免全表掃描）

-- 1. 滑動探索：排除已滑過的用戶
CREATE INDEX idx_swipes_from_user ON swipes(from_user_id, created_at DESC);

-- 2. 配對查詢：找出互相 Like 的組合
CREATE INDEX idx_swipes_direction ON swipes(to_user_id, direction)
  WHERE direction = 'like';

-- 3. 審核後台：按狀態篩選申請
CREATE INDEX idx_applications_status ON applications(status, created_at DESC);

-- 4. 聊天訊息：按配對 ID + 時間排序
CREATE INDEX idx_messages_match ON messages(match_id, created_at ASC);
```

#### 照片載入策略

```
方案：三層載入
  1. Blurhash（模糊佔位，< 30 bytes，存在 DB）
     └─ 卡片出現的瞬間就有視覺，感知速度極快
  2. 縮圖（300px，Supabase Storage Transform）
     └─ 用於卡片列表
  3. 原圖（卡片點開查看全照時）
     └─ 按需載入，不預載

Flutter 套件：
  cached_network_image   ← 自動快取，避免重複下載
  blurhash_dart          ← Blurhash 解碼
```

#### 分頁策略：Cursor-based（游標分頁）

```
❌ 不用 Offset 分頁（第 20 頁之後效能急劇下降）
✅ 用 Cursor 分頁（效能穩定，無論幾頁）

Supabase 實作：
  .from('profiles')
  .select()
  .lt('created_at', lastSeenCreatedAt)  ← cursor
  .order('created_at', ascending: false)
  .limit(10)
```

#### 聊天訊息優化

```
問題：直接訂閱所有訊息會造成不必要的 Realtime 連線

解法：
  - 只在聊天頁面開啟 Realtime 訂閱
  - 離開聊天頁面立即 unsubscribe
  - 首次進入聊天室：一次性 query 最後 50 筆
  - 之後：只透過 Realtime 接收新訊息

Supabase Realtime channel 管理：
  進入聊天 → channel.subscribe()
  離開聊天 → channel.unsubscribe()
  App 進背景 → 暫停所有 Realtime（AppLifecycleState）
```

#### 圖片上傳優化（申請審核照片）

```
上傳前在 client 端壓縮：
  最大尺寸：1200px（長邊）
  壓縮品質：85%
  格式：JPEG（比 PNG 小 60–70%）

Flutter 套件：flutter_image_compress
效果：原本 4MB 的手機照片 → 壓縮後約 300–500KB
      節省 Storage 費用，也加快審核後台載入速度
```

#### 不要過度優化的部分

```
MVP 階段先不處理：
  - CDN 多節點快取（Supabase 內建已足夠）
  - 訊息的已讀回執（read receipt）太複雜，先不做
  - 離線模式（交友 App 不需要）
  - 複雜的配對演算法（先用時間順序，夠用了）
```

## 12. 法律合規框架

> ⚠️ 以下分析僅供參考，正式推出前請諮詢台灣專業法律顧問。

### 台灣法律環境評估

台灣目前**沒有專門禁止「外貌商業篩選」的法律**。
最關鍵的區分：

| 情境 | 法律保護 | 備註 |
|------|----------|------|
| 就業歧視（外貌） | ✅ 受《就業服務法》規範 | 不適用於本平台 |
| 商業服務入會篩選 | ⚠️ 法律灰色地帶 | 可防守，需謹慎設計 |
| 個人資料蒐集 | ✅ 受《個資法》規範 | **最實際的法律風險** |

---

### 主要法律風險與防守策略

#### 風險一：消費者保護法

```
攻擊角度：「你收了我的資料卻不讓我用，是不公平交易」

防守關鍵：
✅ 申請本身完全免費（免費申請被拒不構成消費糾紛）
✅ 申請前明確揭露審核機制（ToS 需清楚說明）
✅ ToS 加入：「申請不代表保證通過，平台保留最終審核權利」
```

#### 風險二：公平交易法

```
攻擊角度：「審核標準不透明，構成不公正商業行為」

防守關鍵：
✅ 不需要公開評分細則，但需有內部一致的書面 SOP
✅ 拒絕通知只說「不符合社群標準」，不說具體原因
✅ 留存審核記錄，萬一被質疑時可自證一致性
```

#### 風險三：名譽損害 / 精神損害

```
攻擊角度：「你說我醜，造成我精神損害」（情緒最大，法律最弱）

防守關鍵：
✅ 永遠不說「因為你不夠好看」或任何主動貶低的語言
✅ 只說「暫時未符合社群標準，30天後可重新申請」
✅ 提供申訴管道，降低用戶的激烈反應

❌ 絕對不能說：
   - 「你的外貌不符合要求」
   - 「你不夠好看」
   - 任何涉及外貌的具體評語
```

#### 風險四：個人資料保護法（最實際的風險）

```
核心問題：你在審核過程中蒐集了「人臉照片」
→ 屬於生物特徵資料，《個資法》有明確規範

必做事項：
✅ 明確告知照片蒐集目的（僅用於入會審核）
✅ 審核完畢後，未通過者照片需有明確刪除機制
✅ 通過者照片的使用範圍需在 ToS 內說明
✅ 不能將審核照片用於任何其他目的（如訓練 AI 等）
```

```
上線前必須完成：
□ 服務條款（ToS）由律師審閱
  └─ 申請不保證通過條款
  └─ 平台保留審核權利條款
  └─ 個資蒐集目的與刪除政策
  └─ 申訴機制說明

□ 隱私權政策（Privacy Policy）
  └─ 人臉照片的蒐集、使用、刪除說明

  └─ 評分標準文件化

□ 申訴流程設計
  └─ 申訴信箱或表單
```
