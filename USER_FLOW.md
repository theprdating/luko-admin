> **Q: 為什麼不加 Google/Apple 登入？**
> 只要加入任何第三方登入（Google），Apple 就強制要求同時提供 Apple Sign In，增加複雜度。
> 電話 OTP 是最符合 Luko 信任模型的方案（一支手機 = 一個真實帳號），日後有需求再加。

## 1. 整體流程地圖

```
App 啟動
  │
  ▼
[Splash Screen]（500ms，僅顯示 Logo）
  │
  ▼
檢查本地 Session Cache（Supabase Flutter 自動存入 SharedPreferences）
  │
  ├─ 無 Session ─────────────────────────────────→ [歡迎頁]
  │                                                   ├─ 申請加入 → [申請流程]
  │                                                   └─ 登入 ────→ [登入頁]
  │
  └─ 有 Session（驗證 Token）
       │
       ├─ Token 失效 ──────────────────────────────→ 清除 Cache → [歡迎頁]
       │
       └─ Token 有效 → 拉取帳號狀態
            │
            ├─ pending  ────────────────────────→ [等待審核頁]
            │
            ├─ rejected ────────────────────────→ [未通過頁]（可倒數重申請）
            │
            └─ approved
                 │
                 ▼
                 條款版本比對
                 （profiles.terms_accepted_at vs app_config.terms_updated_at）
                 │
                 ├─ 條款已是最新 ──────────────→ [主 App]
                 │                                  ├─ 探索（滑卡）
                 │                                  ├─ 配對列表
                 │                                  ├─ 訊息（聊天）
                 │                                  └─ 個人檔案
                 │
                 └─ 條款有更新
                      ▼
                      [強制條款更新 Modal]（無法跳過）
                      ├─ 同意 → 更新 DB + Cache → [主 App]
                      └─ 拒絕 → 登出 → [歡迎頁]
```

**Cache 策略（SharedPreferences）：**

| 快取 Key | 內容 | 失效時機 |
|---------|------|---------|
| `supabase.auth.token` | JWT + Refresh Token | Supabase 自動管理 |
| `luko.account_status` | `pending` / `approved` / `rejected` | 收到推播通知、登出 |
| `luko.terms_accepted_at` | 最後接受條款時間（ISO 8601） | 接受新條款後更新、登出 |
| `luko.profile_cache_ts` | 上次拉取 profile 的時間 | 超過 5 分鐘背景刷新 |

**登出流程：**
1. `await supabase.auth.signOut()` → 清除 Supabase session
2. `await SharedPreferences.clear()` → 清除所有 `luko.*` cache
3. `Navigator.pushAndRemoveUntil(WelcomePage)` → 清除整個 navigation stack

## 2. 新用戶流程

### 2-1. 申請流程（5 步驟）

```
[歡迎頁]
  │
  └─ 點「申請加入」
       │
       ▼
[Step 1｜手機號碼驗證]
  - 輸入手機號碼（含國碼選擇）
  - 發送 OTP 簡訊
  - 輸入 6 位數驗證碼
  - 驗證成功 → 建立 Supabase Auth 帳號
       │
       ▼
[Step 2｜基本資料]
  - 顯示名稱（必填）
  - 生日（必填，需 18 歲以上）
  - 性別（必填：男 / 女 / 其他）
       │
       ▼
[Step 3｜照片上傳]
  - 上傳 2–3 張近期照片
  - 規定：正臉清晰、近期拍攝、真實本人
  - 可預覽、重選、排序
  - 上傳至 Supabase Storage（client 端先壓縮至 ≤ 800KB）
       │
       ▼
[Step 4｜自我介紹（選填）]
  - 簡短文字介紹（上限 150 字）
  - 可略過
       │
       ▼
[Step 5｜確認送出]
  - 預覽填寫資料
  - 勾選同意服務條款 + 隱私權政策（必選）
  - 點「送出申請」
      → 記錄 applications.terms_accepted_at = NOW()
      → 記錄 applications.privacy_accepted_at = NOW()
      → 寫入 DB，狀態設為 pending
  ⚠️ 注意：此處是「首次接受」，不是「版本比對」
      版本比對只針對 approved 用戶的回訪（見第 1 節啟動流程）
       │
       ▼
[等待審核頁]
```

---

### 2-2. 等待審核頁

```
顯示內容：
  ✅ 申請已收到
  ⏳ 審核時間：24–48 小時
  📧 結果將以 Email 通知

用戶可做的事：
  - 查看已上傳的照片（只讀）
  - 等待通知

App 進入 → 自動偵測狀態：
  - 仍 pending → 繼續顯示等待頁
  - 變 approved → 自動跳轉到「歡迎進入」頁
  - 變 rejected → 跳轉到「未通過」頁
```

---

### 2-3. 審核通過流程

```
[Email / SMS 通知：恭喜通過]
  │
  └─ 用戶點通知 or 打開 App
       │
       ▼
[歡迎進入頁]（一次性動畫頁）
  - 顯示「Welcome to Luko」
  - 說明接下來如何使用
  - 點「開始」→ 進入主 App [探索頁]
```

---

### 2-4. 審核未通過流程

```
[Email / SMS 通知：審核結果]
  │
  └─ 用戶打開 App
       │
       ▼
[未通過頁]
  顯示：
    「感謝您申請加入 Luko。
     您目前暫時未符合我們的社群標準。
     您可以在 [日期] 後重新申請。」

  倒數計時：顯示距離可重申請的天數

  ⚠️ 注意：
    - 絕對不說具體原因
    - 不說「顏值」「不夠好看」等字眼
    - 語氣保持中性、不帶評判

  30 天後：
    - 出現「重新申請」按鈕
    - 點擊 → 回到 [申請流程 Step 3]（照片重新上傳）
```

---

### 2-5. 登入流程（回訪用戶）

```
[歡迎頁]
  │
  └─ 點「登入」
       │
       ▼
[登入頁]
  - 輸入手機號碼
  - 發送 OTP（不論帳號存不存在都發，防止帳號枚舉攻擊）
  - 輸入 6 位數驗證碼
  - 驗證成功 → 由 Supabase Auth 判斷身份
       │
       ├─ 此號碼無帳號 → 視為新用戶，跳至 [申請 Step 2]
       │
       └─ 此號碼有帳號 → 檢查狀態
            ├─ pending  → 等待審核頁
            ├─ rejected → 未通過頁
            └─ approved → 條款版本比對 → 進入主 App
```

> **Rate Limiting**（Supabase Dashboard 設定）：
> 同一號碼每小時最多 3 次 OTP 請求；同一 IP 每小時最多 10 次

### 3-2. 探索頁（滑卡配對）

```
[探索頁]
  │
  ├─ 有待探索的用戶
  │    │
  │    ├─ 右滑 / 點 Like 按鈕
  │    │    ├─ 對方也 Like 過我 → [配對成功頁]
  │    │    └─ 對方還沒滑 → 繼續下一張
  │    │
  │    └─ 左滑 / 點 Pass 按鈕 → 繼續下一張
  │
  └─ 已看完所有人
       └─ [空狀態頁]「已看完所有人，稍後再來」

預載策略：
  - 畫面上顯示 1 張（當前）
  - 背景預載 4 張（Queue）
  - queue < 3 張時，自動背景 fetch 補充
```

### 3-3. 配對成功頁

```
[配對成功頁]（全螢幕 Modal）
```

### 3-5. 聊天室

```
       │
       ├─ 傳送文字訊息
       │
       └─ 右上角「⋯」選單
```

## 4. 帳號管理流程

### 4-1. 設定頁

```
[設定頁]
  │
  ├─ 語言切換（繁中 / English）→ 即時生效
  ├─ 隱私權政策 → 開啟 WebView
  ├─ 服務條款 → 開啟 WebView
  ├─ 聯絡客服（Email）
  ├─ 登出 → 確認 Dialog → 回 [歡迎頁]
  └─ 刪除帳號（⚠️ Apple 強制要求）
```

---

### 4-2. 帳號刪除流程（Apple 強制）

```
[設定頁] → 點「刪除帳號」
  │
  ▼
[確認 Dialog]
  「刪除帳號後，您的所有資料、配對及訊息將永久無法復原。」
  [取消] [確認刪除]
  │
  └─ 確認刪除
       │
       ▼
  後端執行：
    1. Supabase Auth 刪除用戶
    2. profiles 資料標記刪除（soft delete）
    3. 照片從 Storage 刪除
    4. 30 天後系統清除相關資料（定期 Edge Function）
       │
       ▼
  回到 [歡迎頁]
```

## 5. 安全機制流程

### 5-1. 檢舉用戶

```
觸發入口：
  - 聊天室右上角選單
  - 對方個人檔案頁右上角選單

[檢舉 Dialog]
  選擇原因：
    ○ 假照片 / 詐騙
    ○ 騷擾或不當言論
    ○ 垃圾訊息
    ○ 其他

  可選填補充說明

  點「送出檢舉」
    → 寫入 reports table
    → 顯示「感謝您的回報，我們將盡快處理。」
    → 對話繼續（不自動封鎖）
```

---

### 5-2. 封鎖用戶

```
觸發入口：
  - 聊天室右上角選單
  - 對方個人檔案頁右上角選單
  - 檢舉完成後可選「順便封鎖」

[封鎖確認 Dialog]
  「封鎖後，對方將無法看到你，你也無法看到對方。」
  [取消] [封鎖]
  │
  └─ 確認
       → 寫入 blocks table
       → 立即雙向隱藏（RLS 即時生效）
       → 現有聊天室標記為封鎖狀態（不刪除，但不可繼續傳訊）
```

## 6. 審核後台流程

> 審核後台為獨立簡易網頁，由 Co-founder 使用

```
[後台登入頁]
  - 輸入管理員 Email + 密碼
  - Supabase Auth 驗證（admin role）
       │
       ▼
[審核儀表板]
  顯示：
    - 待審核數量（醒目）
    - 今日已審核數量
    - 本週通過 / 拒絕比例
       │
       ▼
[待審核列表]
  每筆申請顯示：
    - 申請者照片（3 張並排）
    - 姓名、年齡、性別
    - 申請時間
    - [通過] [拒絕] 按鈕
       │
  ├─ 點 [通過]
  │    → 確認 Dialog
  │    → 更新 DB status → approved
  │    → 觸發 Edge Function 發送通知 Email
  │    → 建立 profiles 資料
  │
  └─ 點 [拒絕]
       → 確認 Dialog
       → 可選填內部備註（不對外顯示）
       → 更新 DB status → rejected
       → 觸發 Edge Function 發送通知 Email
```
